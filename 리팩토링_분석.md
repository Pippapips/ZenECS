## 1. 아키텍처 분석

### 1.1 현재 구조

ZenECS Core는 잘 구조화된 ECS 프레임워크입니다:

- **Kernel**: 다중 월드 관리 및 프레임 틱 조율
- **World**: 엔티티, 컴포넌트, 시스템, 쿼리, 메시징 통합 API
- **Component Pooling**: 타입별 분리된 컴포넌트 풀
- **System Execution**: 단계별 시스템 실행 파이프라인
- **Query System**: 제로 할당 엔티티 쿼리

### 1.2 강점

✅ **Partial Class 분리**: `World` 클래스가 기능별로 잘 분리됨 (16개 partial 파일)  
✅ **DI 통합**: ServiceContainer를 통한 의존성 주입  
✅ **타입 안전성**: 강력한 제네릭 타입 시스템  
✅ **성능 최적화**: 제로 할당 쿼리, ref 반환, 비트셋 활용  
✅ **명확한 책임 분리**: 각 모듈이 단일 책임 원칙 준수

## 2. 리팩토링 제안

### 2.1 🔴 높은 우선순위

#### 2.1.1 Query API 중복 코드 감소 ✅ **완료**

**문제점**:
- `Query<T1>`부터 `Query<T8>`까지 8개의 거의 동일한 오버로드
- 각 오버로드마다 유사한 코드 패턴 반복
- 변수명 불일치 (`a`, `b`, `c`, `f6`, `g`, `h` 등)

**리팩토링 내용**:
1. ✅ **변수명 일관성 개선**: 모든 변수를 `pool1`, `pool2`, ... `pool8`로 통일
2. ✅ **변수명 의미 명확화**: `rf` → `resolvedFilter`로 변경
3. ✅ **구현 노트 추가**: 왜 각 오버로드가 필요한지 설명하는 주석 추가

**리팩토링 후 코드 패턴**:
```csharp
public QueryEnumerable<T1> Query<T1>(Filter f = default) where T1 : struct
{
    var pool = _componentPoolRepository.TryGetPool<T1>();
    var resolvedFilter = ResolveFilter(f);
    var ctx = new QueryCtx1<T1>(pool, resolvedFilter);
    return new QueryEnumerable<T1>(this, in ctx);
}

public QueryEnumerable<T1, T2> Query<T1, T2>(Filter f = default)
    where T1 : struct where T2 : struct
{
    var pool1 = _componentPoolRepository.TryGetPool<T1>();
    var pool2 = _componentPoolRepository.TryGetPool<T2>();
    var resolvedFilter = ResolveFilter(f);
    var ctx = new QueryCtx2<T1, T2>(pool1, pool2, resolvedFilter);
    return new QueryEnumerable<T1, T2>(this, in ctx);
}
```

**제한 사항**:
- C# 제네릭 제약으로 인해 완전한 통합은 불가능
- 각 `QueryCtx`와 `QueryEnumerable`이 서로 다른 타입이므로 오버로드는 유지 필요
- 소스 생성기 도입은 향후 고려 사항으로 남김

**개선 효과**:
- ✅ 코드 가독성 향상 (변수명 일관성)
- ✅ 유지보수성 향상 (명확한 패턴)
- ✅ 구현 의도 명확화 (주석 추가)

**우선순위**: ✅ 완료
